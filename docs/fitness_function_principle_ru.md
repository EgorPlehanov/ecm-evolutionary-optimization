# Принцип расчета fitness-функции в проекте

Этот документ объясняет, как в `ecm_optimizer` считается целевая fitness-метрика для пары параметров ECM `(B1, B2)`.

---

## 1) Общая идея

Fitness нужна для сравнения кандидатов `(B1, B2)` в оптимизаторах (`DE/RS/PSO/BO/GA`).

Метрика **минимизируется**: чем меньше значение, тем лучше кандидат.

В текущей реализации fitness — это **композит** из трех компонентов:
1. штраф за недобор по вероятности успеха,
2. среднее время до исхода,
3. среднее число кривых до исхода.

---

## 2) Как считается оценка на одном числе `n`

Для фиксированного `n` выполняется `repeats_per_n` повторов.

Внутри каждого повтора запускаются ECM-кривые по одной (до `max_curves_per_n`) по схеме **stop-on-success**:
- если на очередной кривой найден фактор (`success=True`) — повтор сразу останавливается;
- если успеха нет, продолжаем до лимита кривых.

По каждому повтору собирается:
- `success` (был ли успех),
- `curves` (сколько кривых реально понадобилось),
- `seconds` (сколько времени занял повтор).

Затем по числу `n` агрегируются:
- `success_runs` из `runs=repeats_per_n`,
- `total_curves`,
- `total_seconds`.

Из этого получаются производные метрики:
- `success_rate = success_runs / runs`,
- `avg_curves = total_curves / runs`,
- `avg_time = total_seconds / runs`.

---

## 3) Агрегация по набору чисел

Если в датасете несколько чисел, для каждого `n` считается отдельный `EvaluationResult`, затем берутся средние по датасету:
- `mean_success_rate`,
- `mean_curves`,
- `mean_time`.

При `workers > 1` расчеты по разным `n` выполняются параллельно через `ProcessPoolExecutor`.

---

## 4) Формула fitness

В коде используются константы:
- `TARGET_SUCCESS_RATE = 0.90`
- `SUCCESS_SHORTFALL_WEIGHT = 200000.0`
- `TIME_WEIGHT = 10000.0`
- `CURVE_WEIGHT = 50.0`

Сначала считается недобор успеха:

`success_shortfall = max(0, TARGET_SUCCESS_RATE - mean_success_rate)`

Итоговый score:

`score = success_shortfall * SUCCESS_SHORTFALL_WEIGHT + mean_time * TIME_WEIGHT + mean_curves * CURVE_WEIGHT`

Оптимизатор минимизирует `score`.

---

## 5) Интерпретация весов и приоритетов

Из формулы следует приоритет:
1. **Сначала** критично удержать `mean_success_rate` не ниже 0.90 (иначе включается крупный штраф).
2. Если порог достигнут, сильнее влияет время (`TIME_WEIGHT=10000`).
3. Число кривых — дополнительный, более мягкий критерий (`CURVE_WEIGHT=50`).

Практически это означает:
- «быстрый, но слишком ненадежный» кандидат будет сильно наказан;
- среди надежных кандидатов преимущество у более быстрых;
- при близком времени выигрывает вариант с меньшим числом кривых.

---

## 6) Что возвращают функции

### `fitness_with_stats(...)`
Возвращает:
- `score` (float),
- `stats` (словарь с агрегатами: `mean_success_rate`, `mean_time_to_outcome_sec`, `mean_curves_to_outcome`, и др.).

Дополнительно:
- для одного числа добавляется `raw_runs`;
- для набора чисел всегда есть `raw_runs_by_n` (детали повторов по каждому `n`).

### `fitness_composite(...)`
Упрощенная обертка: возвращает только `score` без расширенной статистики.

---

## 7) Почему такая схема удобна

- Сочетает надежность и производительность в одной целевой функции.
- Поддерживает раннюю остановку в повторе (реалистично для ECM-практики).
- Дает единый scalar-score для любых оптимизаторов.
- Сохраняет диагностику (`raw_runs`), чтобы анализировать поведение после оптимизации.
