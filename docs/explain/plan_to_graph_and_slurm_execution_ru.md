# Как план превращается в граф и как задачи запускаются параллельно в СКЦ

Ниже — практическое пояснение для отчета: как JSON-план (`ecm_run_plan_v1`) в нашем проекте материализуется в граф зависимостей и как этот граф исполняется через Slurm на СКЦ.

---

## 1) Что является «узлами графа»

Вход — файл плана (например, из `data/plans/*.json`) с полем `operations`.

- Каждый элемент `operations` с `type` (`generate|optimize|validate|analyze`) становится **узлом**.
- Блоки `repeat` не являются «вычислительным» узлом сами по себе — они **разворачиваются** в несколько узлов (по числу итераций).
- После разворачивания (`materialization`) у каждого шага есть:
  - `index` (порядковый номер),
  - `label` (уникальная метка шага),
  - `type`,
  - `args`,
  - `depends_on` (явные зависимости по label).

Именно этот список материализованных операций и есть основа для DAG.

---

## 2) Как появляется граф зависимостей (ребра)

Ребра строятся из зависимостей `depends_on` и частично из ссылок в аргументах:

1. **Явные зависимости**
   - Если у шага есть `depends_on`, то создаются ребра от указанных label к текущему шагу.

2. **Проверка корректности DAG**
   - label должны быть уникальны;
   - зависимость должна ссылаться на существующий предыдущий шаг;
   - запрещены «зависимости вперед» (только на шаги с меньшим `index`);
   - действует типовая валидация:
     - `optimize` может зависеть от `generate`,
     - `validate` — от `optimize` или `generate`,
     - `analyze` — от `validate`/`optimize`/`generate`.

3. **Неявные зависимости по ссылкам (`$ref`)**
   - Для slurm-оркестратора дополнительно анализируются «неразрешенные» ссылки в `args` (маркеры вида `<unresolved:...>`).
   - Это позволяет понять, что шаг логически опирается на результат шага с определенным label, даже если `depends_on` не прописан явно.

4. **Спец-логика для `analyze`**
   - Для `analyze` добавляются зависимости от предыдущих `optimize/validate`, если вход `analyze` совпадает/пересекается по ref-токенам с их dataset.
   - Это предотвращает ранний запуск `analyze` до готовности нужных артефактов.

Результат: ориентированный ациклический граф (DAG), где вершины — шаги плана, ребра — зависимости.

---

## 3) Как запускается параллелизм на СКЦ (Slurm)

Команда `run-plan-slurm` выполняет оркестрацию «один узел DAG = одна Slurm job»:

1. План разворачивается в материализованный список шагов.
2. Для каждого шага формируется `sbatch`-команда.
3. Если у шага есть зависимости, они превращаются в:
   - `--dependency=afterok:<jobid1>:<jobid2>:...`
4. В Slurm это означает:
   - шаг стартует **только после успешного завершения** всех родительских job.

Отсюда параллельность получается «автоматически»:
- все шаги без взаимных зависимостей могут идти одновременно;
- шаги, которые зависят друг от друга, выстраиваются в цепочки.

Для защиты от некорректной цепочки используется `--kill-on-invalid-dep=yes`.

---

## 4) Роль `run-plan-slurm-step` и общего state

Каждый `sbatch` запускает внутреннюю команду `run-plan-slurm-step` для конкретного `step-index`.

Что происходит внутри:
1. Читается исходный план.
2. Читается общий state-файл (по умолчанию `slurm_logs/plan_state.json`).
3. Выполняется только один материализованный шаг.
4. Выход шага парсится (`key=value`) и сохраняется в state под `label` шага.

Зачем нужен state:
- следующие шаги могут ссылаться на результаты прошлых через `$ref`;
- это «склейка» независимых Slurm job в единый логический pipeline.

Чтобы несколько job корректно писали в один state-файл, используется файловая блокировка (`fcntl.flock`).

---

## 5) Как это выглядит на примере полного плана по методам

Типичный шаблон:
1. `generate` (создать датасет),
2. `repeat(method)`:
   - `optimize(method)`
   - `validate(method)`
3. `analyze` (сводная аналитика).

После разворачивания `repeat` получаем отдельные ветки для `de/rs/pso/bo/ga`.

- Все `optimize_*` обычно могут стартовать параллельно после `generate`.
- Каждый `validate_*` ждет свой `optimize_*`.
- Финальный `analyze` ждет завершения веток, от которых зависит.

Именно это и есть DAG-исполнение на СКЦ: максимум параллельности при соблюдении зависимостей.

---

## 6) Что важно указать в отчете

Коротко:
- План сначала **материализуется** (разворачиваются `repeat`, резолвятся ссылки/аргументы).
- Далее строится **DAG зависимостей** по `depends_on` + inferred deps из ссылок.
- Оркестратор `run-plan-slurm` отправляет шаги в Slurm как отдельные job.
- Порядок обеспечивается `--dependency=afterok`, а параллелизм возникает между независимыми ветками.
- Результаты шагов передаются между job через общий `plan_state.json`.

Это объясняет, как «JSON-план» превращается в «граф работ» и затем исполняется как зависимый параллельный workflow на СКЦ.
