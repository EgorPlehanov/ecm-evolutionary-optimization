# Пояснение по SLURM-скриптам `run_ecm_plan.slurm` и `run_ecm_plan_multinode.slurm`

Ниже — краткое описание двух рабочих сценариев запуска плана ECM на СКЦ: «обычный одиночный запуск» и «оркестрация через Slurm-зависимости».

---

## 1) `run_ecm_plan.slurm` — простой запуск плана в одной job

Этот скрипт нужен, когда вы хотите выполнить план целиком внутри **одной** Slurm-задачи.

### Что делает по шагам
1. Запрашивает ресурсы (`#SBATCH`):
   - 1 узел, 1 task, фиксированное число CPU,
   - partition/time/job-name,
   - файлы `stdout/stderr`.
2. Активирует окружение (модули/venv, если настроены в скрипте).
3. Экспортирует `OMP_NUM_THREADS="${SLURM_CPUS_PER_TASK}"`.
4. Берет имя плана из `PLAN_NAME` (или дефолт).
5. Запускает:
   - `srun ecm-optimizer run-plan --plan "${PLAN_NAME}"`.

### Когда использовать
- Для отладки и простых прогонов;
- когда удобно, чтобы весь pipeline шел последовательно в одном job-логе;
- когда не нужна тонкая внешняя оркестрация зависимостей между шагами.

### Плюсы
- Максимально простой запуск;
- один job-id и единый лог.

### Ограничения
- Внешний параллелизм Slurm по шагам плана не используется (вся логика внутри одного процесса `run-plan`).

---

## 2) `run_ecm_plan_multinode.slurm` — оркестрация плана как графа job

Этот скрипт нужен для режима, где шаги плана отправляются как отдельные Slurm job с зависимостями.

Ключевая команда внутри — `ecm-optimizer run-plan-slurm`.

### Что делает по шагам
1. Запрашивает ресурсы для job-оркестратора (обычно небольшие).
2. Экспортирует `OMP_NUM_THREADS`.
3. Берет `PLAN_NAME` из переменной окружения или дефолта.
4. Делает проверку плана (dry-run):
   - `ecm-optimizer run-plan-slurm --plan "${PLAN_NAME}" --dry-run ...`
5. Запускает оркестратор:
   - `srun ... ecm-optimizer run-plan-slurm --plan "${PLAN_NAME}" ...`

### Что делает `run-plan-slurm` под капотом
- Материализует `operations` (включая `repeat`) в список шагов;
- строит зависимости между шагами;
- для каждого шага формирует `sbatch`;
- проставляет `--dependency=afterok:<jobids>`;
- сохраняет/читает общий `plan_state.json` для передачи результатов между шагами.

### Когда использовать
- Для длинных/тяжелых прогонов;
- когда важно распараллелить независимые ветки плана;
- когда нужна явная зависимая оркестрация на уровне Slurm.

### Плюсы
- Эффективный параллелизм между независимыми шагами;
- прозрачные зависимости и отдельные job-логи;
- лучше масштабируется для больших планов.

### Ограничения
- Больше служебной сложности (state-файл, больше job, больше логов).

---

## 3) Главное отличие между скриптами

- `run_ecm_plan.slurm`:
  - **одна Slurm job**;
  - внутри нее выполняется `run-plan`.

- `run_ecm_plan_multinode.slurm`:
  - **одна job-оркестратор + множество дочерних sbatch job**;
  - зависимости управляются через `afterok`.

Иными словами: первый вариант — «монолитный запуск», второй — «DAG-оркестрация в Slurm».

---

## 4) Как запускать (типовые примеры)

### Обычный одиночный запуск
```bash
sbatch --export=PLAN_NAME=full_all_methods_single_run_20d run_ecm_plan.slurm
```

### Оркестрация через Slurm-зависимости
```bash
sbatch --export=PLAN_NAME=full_all_methods_single_run_20d run_ecm_plan_multinode.slurm
```

Если `PLAN_NAME` без `.json`, CLI ищет файл в `data/plans/`.

---

## 5) Что указать в отчете

Короткая формулировка:
- В проекте есть два контура запуска:
  1) последовательный (`run-plan`) в одной job,
  2) графовый (`run-plan-slurm`) с разбиением на job и зависимостями.
- Для нагрузочных прогонов на СКЦ рекомендуется `run_ecm_plan_multinode.slurm`, т.к. он лучше использует параллелизм независимых шагов плана.
