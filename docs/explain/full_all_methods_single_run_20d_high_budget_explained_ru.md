# Краткое пояснение плана `data/plans/full_all_methods_single_run_20d_high_budget.json`

Этот файл — **сценарий полного эксперимента ECM** в формате `ecm_run_plan_v1`.
Он нужен, чтобы **одним запуском**:
1) сгенерировать датасет для 20-значных делителей,
2) прогнать оптимизацию для всех 5 методов,
3) провалидировать найденные параметры,
4) собрать общую аналитику.

---

## Что в нем находится

## 1. Блок верхнего уровня
- `format`: версия формата плана (`ecm_run_plan_v1`).
- `description`: человеко-читаемое описание, что это high-budget конфигурация.
- `params`: общие параметры, которые переиспользуются в нескольких шагах.
- `operations`: последовательность операций (что делать и в каком порядке).

## 2. `params` — общие настройки

### `seed`
Фиксирует случайность (`42`), чтобы запуск можно было воспроизводить.

### `search`
Границы поиска для `B1/B2`:
- `b1-min`, `b1-max`
- `b2-min`, `b2-max`
- `ratio-max` (ограничение на отношение `B2/B1`)

Это общий диапазон, в котором методы ищут хорошие параметры.

### `method_list`
Список методов, которые будут запущены по очереди:
- `de`, `rs`, `pso`, `bo`, `ga`.

### `method_opt`
Индивидуальные параметры **оптимизации** для каждого метода.
Пример:
- у DE: `de-popsize`, `de-maxiter`
- у BO: `bo-initial-samples`, `bo-iterations`, `bo-candidate-pool`
- у GA: `ga-population-size`, `ga-generations`, `ga-mutation-prob`

То есть здесь собраны специфичные ручки тюнинга каждого алгоритма.

### `shared_opt_args`
Общие параметры оптимизации для всех методов:
- `max-curves-per-n`: бюджет кривых на число,
- `repeats-per-n`: количество повторов на число.

### `shared_val_args`
Общие параметры валидации (обычно больше бюджета, чем в optimize):
- `max-curves-per-n`
- `repeats-per-n`

### `generate_args`
Параметры генерации датасета:
- `target-digits`: размер делителя,
- `cofactor-digits`: размер кофактора,
- `train-count`: размер train,
- `control-count`: размер control.

### `analyze_group_by`
Как группировать итоговую аналитику (в этом плане: по `divisor_size` и `method`).

---

## Как устроен блок `operations`

План выполняется в 3 крупных шага.

### Шаг 1: `generate`
Операция:
- `type: generate`
- `label: dataset_20_single`

Создает датасет и сохраняет его путь в артефакты запуска.

### Шаг 2: `repeat` по всем методам
Используется цикл:
- `repeat.as = method_iter`
- `repeat.values.method = $ref:params.method_list`

Для каждого метода внутри цикла выполняются **две операции**:

1) `optimize`:
- получает dataset из шага generate,
- получает `method`, `seed`,
- объединяет (`$spread_ref`) три набора аргументов:
  - `params.shared_opt_args`
  - `params.method_opt.<текущий_метод>`
  - `params.search`

2) `validate`:
- берет тот же dataset,
- берет файл результата оптимизации через ссылку:
  `opt-result-file: $ref:opt_<method>_single.result_file`,
- применяет `params.shared_val_args`.

Итог: для каждого метода есть пара «оптимизация + валидация» на одном и том же датасете.

### Шаг 3: `analyze`
Операция:
- `type: analyze`
- `label: analysis_single_dataset_all_methods`

Собирает итоговую сводку по всем методам на сгенерированном датасете.

---

## Что означают `$ref` и `$spread_ref`

- `$ref`: подставить значение из другого места плана или из артефакта предыдущего шага.
  - пример: `dataset: $ref:dataset_20_single.dataset_dir`
- `$spread_ref`: «развернуть» набор параметров в текущий `args`.
  - полезно, чтобы не дублировать одинаковые поля вручную.

Это делает JSON компактным и удобным для поддержки.

---

## Зачем нужен именно этот high-budget план

Это конфигурация для **более надежной оценки** по сравнению с легкими/быстрыми прогонами:
- повышенные бюджеты в optimize/validate,
- расширенный датасет,
- единая схема сравнения всех 5 методов в одном пайплайне.

Подходит для случаев, когда важнее качество и стабильность метрик, чем минимальное время запуска.
