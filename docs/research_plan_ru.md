# План НИР: эволюционный подбор параметров ECM

## 1. Гипотеза и цель

**Гипотеза:** для фиксированной вычислительной платформы и класса составных чисел существуют пары `(B1, B2)`, отличные от рекомендаций GMP-ECM, которые уменьшают эмпирическое ожидаемое время нахождения делителя заданного размера.

**Цель:** построить автоматизированный pipeline подбора параметров ECM эволюционными/эвристическими методами и сравнить с baseline-таблицами GMP-ECM.

## 2. Экспериментальный дизайн

- Целевые размеры делителей: 30, 35, 40, 45, 50 цифр.
- Для каждого размера:
  - train-набор: 20 чисел;
  - control-набор: 20 чисел.
- Каждое число имеет известный простой делитель целевого размера.

## 3. Фитнес-функция

Для фиксированных `(B1, B2)` и числа `N` выполняются `M` запусков одной кривой ECM.

- `s` — число успешных запусков;
- `T_total` — суммарное время всех запусков.

Оценка:

```text
ET_hat = T_total / s, если s > 0
ET_hat = +inf, если s = 0
```

Финальный fitness по набору из `L` чисел:

```text
F(B1, B2) = (1/L) * sum_j ET_hat_j
```

## 4. Актуальное окно поиска и бюджеты (после 2-го полного прогона)

На основании двух запусков от **30 марта 2026 года**:
- полезные решения устойчиво концентрируются при низких `B1`;
- во многих лучших точках наблюдается `B2/B1` близко к 1–10;
- слишком широкие границы сильно ухудшают эффективность поиска.

### 4.1. Рекомендуемые общие границы (следующий полный прогон)

- `b1_min=1000`
- `b1_max=30000`
- `b2_min=1000`
- `b2_max=600000`
- `ratio_max=20`

### 4.2. Рекомендуемые бюджеты измерений

- optimize: `curves-per-n=12`
- validate: `curves-per-n=24`

## 5. Настройки оптимизаторов (следующий прогон)

Приоритеты по итогам двух прогонов:
- приоритет 1: **PSO, GA, DE**;
- приоритет 2: **RS**;
- приоритет 3: **BO**.

Рекомендуемые параметры:

- **PSO:** `swarm_size=18`, `iterations=12`.
- **GA:** `population_size=22`, `generations=14`, `mutation_prob=0.14`.
- **DE:** `popsize=10`, `maxiter=10`.
- **RS:** `budget=60`.
- **BO:** `initial_samples=16`, `iterations=22`, `candidate_pool=2500`.

## 6. Валидация и статистика

Сравнение на контрольном наборе:
- optimized `(B1*, B2*)`;
- baseline `(B1_tab, B2_tab)`.

Метрики:
- среднее/медиана `ET_hat`;
- относительный выигрыш `%`;
- критерий Манна–Уитни (односторонний);
- оценка дисперсии (IQR/квантили) для выбора между близкими по среднему методами.

## 7. Правила принятия решений

- Если BO на очередном полном прогоне снова не даёт положительного улучшения на control, переносить его в режим редких sanity-check.
- Если разница между топ-методами (PSO/GA/DE) < 3 п.п. по улучшению, выбирать метод с меньшей дисперсией и меньшим runtime.
- Каждое изменение гиперпараметров фиксировать отдельной ревизией плана с привязкой к датасету и timestamp запуска.

## 8. Риски и меры

- Высокая стоимость вычислений → уменьшать `M` на этапе поиска, увеличивать на финале.
- Шум измерений → повторные прогоны, устойчивые метрики (медиана, IQR).
- Переобучение → отдельный контрольный набор и повторные seed'ы.

## 9. Артефакты эксперимента

Для каждого запуска хранить:
- commit hash кода;
- версию `ecm`;
- параметры оптимизатора;
- train/control datasets;
- лучшую пару `(B1, B2)`;
- подробные результаты валидации;
- графики сходимости и траектории параметров.


## 10. Практический run-plan

Для следующего полного запуска использовать `data/plans/baseline_all_methods.json`, где зафиксированы:
- генерация датасета `train=20`, `control=20`;
- optimize: `curves-per-n=12`; validate: `curves-per-n=24`;
- границы: `b1 in [1000, 30000]`, `b2 in [1000, 600000]`, `ratio_max=20`;
- параметры методов: DE/RS/PSO/BO/GA в соответствии с разделом 5.
