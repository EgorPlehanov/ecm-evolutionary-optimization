## 1. Контекст и цель

### 1.1. Контекст
Текущий модуль анализа уже формирует иерархию артефактов (`overview -> divisor_size -> dataset -> method -> seed`), генерирует markdown-отчёты, CSV-таблицы и базовые графики.

### 1.2. Главная цель модернизации
Сделать модуль анализа инструментом принятия решений, который:
1. наглядно показывает результат и устойчивость методов;
2. формирует статистически корректные выводы;
3. автоматически рекомендует кандидатов для внедрения;
4. даёт воспроизводимый машинно-читаемый контракт артефактов (`manifest.json`).

### 1.3. Ключевой критерий успешности
После выполнения данного ТЗ пользователь должен получать по каждому запуску анализа:
- понятный executive summary;
- артефакты по уровням агрегации с единым стандартом;
- статистические выводы с размером эффекта и доверительными интервалами;
- список решений: `adopt / investigate / reject`.

## 2. Область работ (scope)

### 2.1. Входит в scope
- расширение набора таблиц/графиков/статистики на уровнях `overview`, `divisor_size`, `dataset`, `method`;
- модуль статистики (`stats.py`) и интеграция результатов в отчёты;
- улучшение markdown-отчётов (в т.ч. executive summary и навигация);
- полноценный `manifest.json` с деревом артефактов и метаданными;
- контроль качества данных перед анализом (coverage, missing, перекос по seeds и т.д.);
- budget-aware сравнения (при фиксированном времени/итерациях);
- сравнение с предыдущим анализом (regression guard, delta).

### 2.2. Не входит в scope (на текущий этап)
- разработка отдельного веб-интерфейса/дашборда;
- изменение алгоритмов оптимизации (GA/DE/PSO/BO/RS);
- изменение формата сырых экспериментальных логов (кроме минимально необходимых полей для анализа).

---

## 3. Принципы проектирования

1. **Decision-first**: каждый уровень отчёта должен отвечать на конкретный управленческий вопрос.
2. **Статистика + практическая значимость**: p-value без effect size и CI не считается достаточным.
3. **Reproducibility by default**: любой вывод должен быть трассируем к исходным данным и параметрам анализа.
4. **Fail-soft**: при частично отсутствующих данных отчёт строится с предупреждениями, а не падает целиком.
5. **Scalable artifacts**: ограничение визуального шума (max 8-10 серий в одном графике).

## 7. Нефункциональные требования

1. Производительность:
   - Генерация анализа для ~150 запусков не более N минут (N зафиксировать после baseline-замера).
2. Надёжность:
   - Ошибки в части данных не должны ломать весь пайплайн.
3. Воспроизводимость:
   - Все ключевые параметры анализа должны фиксироваться в манифесте.
4. Совместимость:
   - Старые отчёты должны оставаться читаемыми.

## 10. Риски и меры

1. **Риск:** ложноположительные выводы из-за множественных сравнений.  
   **Мера:** Holm correction + обязательный effect size.

2. **Риск:** перегруженные графики и плохая читаемость.  
   **Мера:** max-series policy + small-multiples + агрегат `others`.

3. **Риск:** неполные/грязные данные.  
   **Мера:** Data Quality Gates и статус `partial` в manifest.

4. **Риск:** рост времени генерации отчётов.  
   **Мера:** кэширование промежуточных агрегаций и профилирование.

## 14. Приложение: минимальный шаблон decision block

Для каждого `overview/report.md`:

1. **Лидер:** метод X (метрика Y, CI, effect size).
2. **Стабильность:** устойчивость по dataset/divisor_size.
3. **Риск:** где метод деградирует.
4. **Решение:** `adopt / investigate / reject`.
5. **Следующий эксперимент:** какие параметры/бюджет менять.
