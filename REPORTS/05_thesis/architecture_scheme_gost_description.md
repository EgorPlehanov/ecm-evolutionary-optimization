# Описание архитектурной схемы программной системы подбора параметров ECM (для оформления по ГОСТ)

## 1. Назначение схемы

Архитектурная схема предназначена для графического представления структуры программной системы автоматизированного подбора параметров ECM и потоков данных между ее функциональными подсистемами. Схема должна показывать, как из команд пользователя формируется полный вычислительный цикл: подготовка датасета, оптимизация параметров, валидация результатов и выпуск итоговых аналитических материалов.

## 2. Границы системы

На схеме рекомендуется явно выделить границу программного комплекса `ecm_optimizer` как основного объекта проектирования. Внутри границы размещаются подсистемы генерации данных, оптимизации, вычисления fitness, запуска ECM, валидации и анализа. Внешним по отношению к системе элементом является исполняемый модуль GMP-ECM, с которым взаимодействие осуществляется через процессный вызов.

## 3. Состав функциональных блоков

Для оформления схемы следует использовать следующие блоки (наименования желательно сохранить в едином стиле):

1. **Подсистема командного управления (CLI)**  
   Обеспечивает запуск сценариев `generate`, `optimize`, `validate`, `run-plan`, `analyze`, то есть выполняет роль оркестратора.

2. **Подсистема генерации датасетов**  
   Формирует train/control наборы чисел вида \(N = pq\), подготавливает и сохраняет манифесты входных данных.

3. **Подсистема оптимизации**  
   Реализует алгоритмы RS, DE, GA, PSO и BO, генерирует кандидаты \(B_1/B_2\) и передает их на оценку.

4. **Подсистема вычисления целевой функции (Fitness evaluator)**  
   Выполняет оценку кандидатов, управляет повторами, лимитами и правилами завершения серии запусков.

5. **Подсистема запуска ECM (ECM runner)**  
   Выполняет вызов внешнего исполняемого файла GMP-ECM, контролирует timeout, принимает и нормализует результаты прогонов.

6. **Внешний вычислительный модуль GMP-ECM**  
   Является внешним средством факторизации, возвращающим признаки успеха/неуспеха и служебные данные выполнения.

7. **Подсистема валидации**  
   Выполняет независимую проверку найденных параметров на control-наборе, формирует validate-артефакты.

8. **Подсистема хранения результатов (JSON-артефакты)**  
   Накопление структурированных результатов оптимизации и валидации для последующей агрегации.

9. **Подсистема аналитики и отчетности**  
   Агрегирует артефакты, строит итоговые таблицы и визуализации, формирует Markdown/CSV/PNG-материалы.

## 4. Потоки данных и управляющие связи

При построении схемы необходимо отразить следующие направленные связи:

- CLI → Подсистема генерации датасетов;
- CLI → Подсистема оптимизации;
- Подсистема оптимизации → Fitness evaluator;
- Fitness evaluator → ECM runner;
- ECM runner → GMP-ECM;
- Fitness evaluator → JSON-артефакты;
- CLI → Подсистема валидации;
- Подсистема валидации → ECM runner;
- JSON-артефакты → Подсистема аналитики;
- Подсистема валидации → Подсистема аналитики;
- Подсистема аналитики → Выходные отчеты (Markdown, CSV, PNG).

Для соблюдения методологической корректности важно показать, что оптимизаторы не обращаются к GMP-ECM напрямую, а работают только через Fitness evaluator и ECM runner. Это подчеркивает единообразие процедуры оценки и сопоставимость результатов различных методов.

## 5. Рекомендации по компоновке (вид схемы)

Для курсовой/ВКР удобно применять горизонтальную компоновку слева направо:

- слева — блок управления (CLI);
- в центре — вычислительное ядро (генерация, оптимизация, fitness, runner, валидация);
- справа — артефакты и аналитика;
- внешний модуль GMP-ECM размещать вне границы системы, рядом с ECM runner.

Рекомендуется группировать внутренние блоки по подсистемам в виде рамок:

- «Оркестрация и интерфейсы»;
- «Вычислительное ядро»;
- «Хранилище артефактов»;
- «Аналитика и отчетность».

## 6. Требования к оформлению по ГОСТ (практические)

При подготовке рисунка рекомендуется придерживаться общих норм ЕСКД/ГОСТ для текстовых работ:

1. Все блоки — в едином стиле прямоугольников, одинаковая толщина линий рамок и стрелок.
2. Текст в блоках — краткий, терминологически единообразный, без смешения русского и английского в одном термине (допустимо оставить устоявшиеся имена в скобках, например «Вычисление целевой функции (Fitness evaluator)»).
3. Все стрелки — направленные, с однозначно читаемым направлением потока.
4. Тип связи рекомендуется различать графически:
   - сплошная линия — управляющий/вычислительный поток;
   - штриховая линия (опционально) — поток артефактов/файлов.
5. Подрисуночная подпись оформляется как:  
   **«Рисунок 3.1 — Архитектурная схема программной системы автоматизированного подбора параметров ECM»**.
6. В тексте главы перед рисунком должна быть ссылка вида «Архитектура системы представлена на рисунке 3.1», а после рисунка — поясняющий абзац о ключевых потоках и роли валидационного контура.

## 7. Минимальный сценарий, который должна объяснять схема

Схема должна позволять проследить следующий сквозной сценарий:

1. Пользователь запускает сценарий через CLI.
2. Формируются train/control датасеты.
3. Оптимизатор генерирует кандидата \(B_1/B_2\).
4. Fitness evaluator инициирует серию запусков через ECM runner.
5. ECM runner обращается к GMP-ECM и возвращает нормализованный результат.
6. Результаты оптимизации сохраняются в JSON.
7. Найденные параметры проходят независимую валидацию.
8. Аналитический модуль агрегирует optimize/validate артефакты.
9. Формируются итоговые отчеты и графики.

## 9. Пример поясняющего текста под раздел 3.2

«Архитектурная схема отражает модульное разделение системы на контур оркестрации, вычислительное ядро и контур аналитики. Принципиальной особенностью является централизация процедуры оценки кандидатов \(B_1/B_2\) в блоке вычисления fitness, который взаимодействует с внешним GMP-ECM только через подсистему ECM runner. Такое решение обеспечивает единые условия сравнения оптимизационных методов и воспроизводимость результатов за счет фиксированных правил выполнения, регистрации метаданных и последующей независимой валидации на control-наборе.»
